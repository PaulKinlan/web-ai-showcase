MMS-TTS Assamese — end-to-end Assamese speech, one network

This is the Assamese checkpoint of Meta's MMS-TTS family — a distinct model from the Hindi, Tamil and Marathi demos: its own native Assamese (Eastern Nagari) character vocabulary (66 symbols, is_uroman:false — native Assamese, not romanised Latin) and its own Assamese-trained VITS weights. Assamese shares the Bengali-Assamese script but is a separate checkpoint with its own vocab and the characteristic Assamese letters ৰ (ra) and ৱ (wa) that set it apart from Bengali. VITS is a conditional VAE with normalizing flows and a stochastic duration predictor: one end-to-end network from text straight to a waveform, no separate vocoder, and the same Assamese sentence comes out with slightly different rhythm every time.

Why it matters

An Assamese voice that runs entirely on-device: no per-character bill, no text leaving the tab. Assamese is spoken by tens of millions across Assam and the north-east; VITS won't out-act a big expressive model, but it's the model you reach for when you need Assamese speech everywhere, offline, privately.

  • Offline Assamese read-aloud — narrate Assamese articles, forms or messages with no service to sign up for and nothing sent anywhere.
  • Accessibility — a built-in Assamese screen-voice for content, working without a network.
  • Embeddable Assamese voice — bundle an Assamese TTS inside an app or device UI.
  • Natural variation — the stochastic duration predictor keeps repeated Assamese prompts from sounding mechanically identical.

How the API works

MMS-TTS is normally a one-liner with the text-to-speech pipeline. This ONNX export ships model.onnx at the repo root and omits a tokenizer.json, so instead of the high-level pipeline the worker loads the real VitsModel and reproduces the tokeniser exactly — safe here because this checkpoint is is_uroman:false, so its VitsTokenizer is a deterministic character tokeniser (no uroman romanisation): 

import { AutoModel, Tensor } from "@huggingface/transformers";

// vocab.json is byte-identical to canonical facebook/mms-tts-asm
const model = await AutoModel.from_pretrained(
  "naklitechie/mms-tts-as-ONNX", { model_file_name: "model", subfolder: "", dtype: "fp32" });

// Faithful VitsTokenizer: lowercase, map each in-vocab char, interleave blank (id 0).
// VERIFIED byte-identical to transformers.js's real VitsTokenizer on a control language.
const ids = tokenizeAssamese("এটা নেটৱৰ্কে আখৰক পোনপটীয়াকৈ ধ্বনি তৰঙ্গলৈ ৰূপান্তৰ কৰে।");
const input_ids = new Tensor("int64", BigInt64Array.from(ids.map(BigInt)), [1, ids.length]);
const { waveform } = await model({ input_ids, attention_mask });
// waveform.data → Float32Array (mono), 16 kHz. Run again → slightly different (stochastic duration).

The manual tokeniser was checked byte-for-byte against transformers.js's real VitsTokenizer on a control language (Xenova/mms-tts-hin) — identical token ids — so the Assamese tokenisation is faithful, and the waveform is genuine VitsModel output. The Hindi demo uses the plain pipeline() because its export ships the full tokenizer files.
