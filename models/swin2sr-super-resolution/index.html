Swin2SR — upscale an image 2× and reconstruct the detail

Swin2SR is a Swin-Transformer super-resolution model. Give it a small, soft, or compressed picture and it predicts what a 2×-larger, sharper version should look like — inventing plausible edges and texture, not just smoothing. It runs entirely on your device: the image never leaves the tab.

Swin2SR is heavy per pixel, so this page caps the input's long edge to 512 px and processes it in overlapping tiles — bounding memory and giving honest, incremental progress. A bigger upload is shrunk to the cap first (noted below when it happens).

Why it matters

Upscaling used to mean either a fuzzy bicubic stretch or a paid cloud API that saw your images. Swin2SR in the browser makes real, learned super-resolution a local function call — private, free per call, and offline.

  • Sharper thumbnails & product shots — enlarge small catalogue or avatar images to fill a retina layout without a re-shoot or an upscaling service.
  • Rescue compressed images — recover edges lost to heavy JPEG/WebP compression on user uploads before you store or display them.
  • Restore old / low-res photos — bring scanned or early-digital pictures up to a usable size, entirely client-side, on sensitive family photos that should never be uploaded.
  • On-device pipelines — cut a subject out with background removal, then super-res just the subject; or enhance a webcam frame before OCR or detection.

How the API works

Super-resolution is a one-liner with Transformers.js. The output is a RawImage at 2× the input:

import { pipeline } from "@huggingface/transformers";

const upscaler = await pipeline(
  "image-to-image",
  "Xenova/swin2SR-classical-sr-x2-64",
); // downloads + caches ~150 MB once

const output = await upscaler(imageURL);
// → RawImage { data: Uint8Array, width, height, channels }
//   width/height are exactly 2× the input; data is raw RGB pixels

This page goes one step further: instead of one giant forward pass, it splits the input into overlapping tiles, calls the pipeline per tile inside a Web Worker, crops the overlap, and stitches the cores back together — so memory stays bounded and you get real per-tile progress. The "before" layer in the slider is the browser's own bicubic 2× zoom, for an honest comparison.
