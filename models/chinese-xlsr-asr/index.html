wav2vec2-XLSR-53 — Mandarin Chinese speech, one hanzi per frame

The English wav2vec2 demo speaks one language, the Russian XLSR demo another, and the French, Spanish and Italian demos more. This one takes the same cross-lingual cousin somewhere new: XLSR-53 was pre-trained on 53 languages (56,000 hours of multilingual audio) and then fine-tuned on Mandarin Chinese. Its output alphabet is not 26 Latin letters or 33 Cyrillic ones — it is a ~3,500-symbol vocabulary of Chinese characters (hanzi). Same CTC mechanics (label every frame, then collapse), but the model has to pick the right character for each run of sound, with no language model to lean on — so it substitutes homophones the way a phonetic listener would (加/迦, 大/打, 木/目). Pick a clip and watch the frames collapse into Chinese.

Why it matters

Mandarin has roughly a billion speakers, and its writing system makes it a great stress-test for speech AI: an ASR model can't spell out a hanzi phonetically the way it can a Latin word — it has to pick the right character from thousands. Cross-lingual self-supervised pre-training is how XLSR-53 gets there: it learns what human speech sounds like from 53 languages at once, so fine-tuning to Mandarin needs far fewer transcripts. That is what puts a usable Chinese recogniser inside a browser tab — private, offline once cached, no server, no key. And because it's CTC (frames, not generated tokens) it can't hallucinate an unspoken sentence, and every character is pinned to a moment in time.

  • On-device Chinese dictation and captions — for apps that must keep audio on the device (health, legal, journalism) across the Mandarin-speaking world.
  • One backbone, many languages — the same XLSR-53 recipe produces the English, Russian, French, Spanish, Italian and Chinese recognisers in this showcase.
  • Per-character forced alignment — real per-hanzi timings for karaoke, subtitles, and pronunciation feedback, straight from the frames (Chinese has no spaces, so the character is the natural unit).
  • A homophone teaching aid — the model's homophone substitutions (加/迦, 木/目) make the acoustic ambiguity of Mandarin visible.

How the API works

This repo ships the standard transformers.js layout, so we load the model directly — but as AutoModelForCTC (not the ASR pipeline) so we can see the per-frame argmax. Because the character vocab is ~3,500 entries, we build the id→character map from vocab.json and do the CTC collapse ourselves (exactly what the pipeline does internally): 

import { AutoModelForCTC, Tensor } from "@huggingface/transformers";

const model_id = "onnx-community/wav2vec2-large-xlsr-53-chinese-zh-cn-ONNX";
const model = await AutoModelForCTC.from_pretrained(model_id, {
  device: "wasm", dtype: "q4",   // 4-bit export (~244 MB) — coherent Mandarin; fp32 is 1.28 GB
});

// audio = 16 kHz mono Float32Array → normalise to zero mean / unit variance (wav2vec2 feature step)
const x = normalize(audio);
const { logits } = await model({ input_values: new Tensor("float32", x, [1, x.length]) });
// logits shape [1, frames, ~3503]  ← one score per Chinese character per frame

// greedy CTC decode: argmax per frame, merge duplicates, drop the blank (id 0)
const ids  = argmaxPerFrame(logits);    // e.g. [0,0,1520,1520,0,880,880,...]
const text = collapse(ids, id2char);    // "中国…" — no decoder involved

Inference runs in a Web Worker so the UI never janks. The worker returns the raw per-frame argmax and real per-character timings (from the frame each character was emitted at) — nothing is faked or pre-computed.
