Qwen2.5-Coder — a coding model that runs on your GPU

Qwen2.5-Coder-1.5B is a code-specialised instruct model from Alibaba — trained on ~5.5 trillion tokens of code and code-adjacent text — and it runs entirely in your browser, compiled to WebGPU by MLC's WebLLM engine. Unlike a general chat model, it's tuned for programming: write a function from a comment, complete code, generate tests, regex or SQL, and explain snippets — with syntax-highlighted streaming, tokens per second, and the exact prompt template on show. Nothing is uploaded, there's no API bill, and it works offline once cached. WebLLM is WebGPU-only, so it needs a real GPU.

Why it matters

A code model small enough to run in a tab, but tuned for programming, turns "AI code assistance" into a client-side primitive — private by construction, free to run, and available offline once the weights are cached. No snippet you paste ever leaves the device.

  • Private inline assistance — complete a function, write a test, or draft a regex without shipping proprietary code to a cloud API.
  • Learn-by-explaining — paste an unfamiliar snippet and get a plain-English walkthrough, on-device, for students or code review.
  • Boilerplate on tap — SQL from a description, a data-class from a schema, argument parsing — the tedious 80% generated locally.
  • A local codegen stage — pair it with a reasoning model that plans, and let the Coder write the implementation (see the multi-model demo).

How the API works

WebLLM gives you an OpenAI-shaped chat API backed by a WebGPU engine. You create the engine (which downloads and compiles the weights), then stream a completion. There's nothing code-specific in the API — you wrap the user's snippet in an instruction and let the code-tuned weights do the rest. This page uses the shared helpers in lib/webllm.js, running inside a Web Worker so the UI stays responsive: 

import { webGPUAdapterAvailable, createEngine, streamChat }
  from "/web-ai-showcase/lib/webllm.js";

// WebLLM is WebGPU-only — gate before doing anything (headless requestAdapter() → null).
if (!(await webGPUAdapterAvailable())) {
  showNeedsWebGPU();          // never a faked completion
} else {
  const engine = await createEngine({
    model: "Qwen2.5-Coder-1.5B-Instruct-q4f16_1-MLC",
    onProgress: (p) => setProgress(p.text, p.progress),   // ~880 MB, cached after first load
  });

  const messages = [
    { role: "system", content: "You are a coding assistant. Reply with a single fenced code block." },
    { role: "user",   content: "Language: python\n# return the nth Fibonacci number" },
  ];

  const full = await streamChat(
    engine,
    { messages, temperature: 0.2, top_p: 0.95, max_tokens: 512 },
    (delta) => appendToken(delta),          // streams token-by-token on your GPU
  );
  // engine.interruptGenerate() stops decoding early; engine.runtimeStatsText() reports tok/s.
}

Everything here runs exactly that path in worker.js, and gates on a real WebGPU adapter first. No adapter → a labelled needs-WebGPU state, never a canned completion.
