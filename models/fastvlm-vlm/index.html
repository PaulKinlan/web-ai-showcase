FastVLM — ask a fast, small model about a picture

FastVLM is Apple's efficient vision-language model. Its trick is the vision encoder — a hybrid conv/transformer called FastViTHD that emits far fewer, higher-quality visual tokens, so the model starts answering quickly. Give it an image and a question and the 0.5B version answers in plain language, streaming token by token — on your GPU, or on the CPU when there's no GPU, with nothing uploaded.

Why it matters

FastVLM's contribution is speed: a vision encoder designed so the model can start answering about a high-resolution image quickly, at a size (0.5B) that fits in a browser tab — on a GPU or, more unusually for a VLM, on the CPU.

  • Low-latency visual Q&A — point-and-ask experiences with no server round-trip (on the CPU here; a hardware GPU makes it markedly faster).
  • On-device document triage — read a photographed receipt or label and pull out the numbers, privately.
  • Automatic alt-text — describe every image on a page for screen-reader users, at no per-image API cost.
  • Runs without a GPU — the small size means it runs on the CPU (WASM), so it works on devices with no usable GPU.

Use cases

Basics

Describe or ask about an image — the core "talk to a picture" loop.

Practical

Read a receipt or screenshot and extract answers — visual Q&A over documents.

Wild

Interrogate an image — keep asking follow-up questions in an open visual chat.

Multi-model

FastVLM describes an image → an OPUS-MT model translates it: multilingual alt-text.

How the API works

FastVLM loads through the generic image-text-to-text auto classes. You build a chat prompt with an <image> marker, run the image + prompt through the processor, then generate — streaming tokens as they come: 

import { AutoProcessor, AutoModelForImageTextToText, RawImage, TextStreamer }
  from "@huggingface/transformers";

const model_id = "onnx-community/FastVLM-0.5B-ONNX";
const processor = await AutoProcessor.from_pretrained(model_id);
const model = await AutoModelForImageTextToText.from_pretrained(model_id, {
  // all-q4 keeps every IO tensor float32; runs on the WASM CPU backend.
  dtype: { embed_tokens: "q4", vision_encoder: "q4", decoder_model_merged: "q4" },
  device: "wasm",
});

const messages = [{ role: "user", content: "<image>Describe this image in detail." }];
const text = processor.apply_chat_template(messages, { add_generation_prompt: true });
const image = await RawImage.fromURL(imageURL);
const inputs = await processor(image, text, { add_special_tokens: false });

const streamer = new TextStreamer(processor.tokenizer,
  { skip_prompt: true, skip_special_tokens: true, callback_function: (t) => append(t) });
await model.generate({ ...inputs, do_sample: false, max_new_tokens: 120, streamer });

This page runs exactly that in a Web Worker, runs on the WASM CPU backend and reports the real latency + backend, never a fake answer.
