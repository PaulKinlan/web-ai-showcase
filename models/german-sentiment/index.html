German sentiment — a German-native reader of the mood

Most sentiment models are English-first. german-sentiment-bert is the opposite: a bert-base-german-cased fine-tuned on about 1.8 million German samples — reviews, tweets and service messages — to score text as positiv, negativ or neutral. Its whole vocabulary and corpus are German, so it reads compound words, umlauts (ä/ö/ü), the ß and German register as first-class signal. Type something in German below and watch the real class probabilities; then look inside to see which words moved the verdict. Nothing leaves your device.

Why two repos: the weights repo ships an ONNX export but no tokenizer.json (only vocab.txt), which transformers.js can't build a tokenizer from. This model is a fine-tune of bert-base-german-cased, and the two vocabularies are byte-identical, so we load the same WordPiece tokenizer from the base repo and the real weights from the fine-tune — a legitimate AutoTokenizer + AutoModel load.

See inside — which words moved the verdict

The model never matches "sentiment words" from a list — it reads the whole sentence through one German encoder. To make that legible we run occlusion attribution: re-score the text with each word removed and measure how far the winning class's confidence drops (in log-odds). A word tinted in the verdict colour was pushing that verdict; a grey word was pulling against it. Because the tokenizer decides what a "word" contributes, a compound noun, a negation (nicht, kein) or an intensifier all carry real weight.

German-native vs multilingual

A multilingual model (XLM-R) spreads one embedding space across ~100 languages — brilliant when your text could be in any of them. german-sentiment-bert goes the other way: all of its ~110M parameters and its entire vocabulary are German, pre-trained and fine-tuned on German text. On idiomatic German — negation stacking, long compound nouns, service-desk politeness — a specialist often reads the register a generalist flattens. The multi-model demo runs both side by side so you can see where they agree and where they part.

Why it matters

German is the most-spoken native language in the EU, and a lot of opinion data — reviews, DMs, support tickets, social posts — arrives in it. A model that was actually trained on German reads its register, not a translated approximation, and it runs in a tab, so that text never has to leave the device.

  • German review monitoring — triage a feed of German reviews, negatives first, on-device.
  • Private social listening — score German-market sentiment without shipping posts to an API.
  • Community moderation signal — flag a sharp negative turn in a German-language thread.
  • Compound- and negation-aware analytics — trained on German, so long compounds and stacked negations count as the signal they are.

How the API works

The weights repo has no tokenizer.json, so we load the identical tokenizer from the base model and the head + weights from the fine-tune, then run them together:

import {
  AutoTokenizer,
  AutoModelForSequenceClassification,
} from "@huggingface/transformers";

// Same WordPiece vocab as the fine-tune (byte-identical vocab.txt).
const tokenizer = await AutoTokenizer.from_pretrained(
  "google-bert/bert-base-german-cased",
);
const model = await AutoModelForSequenceClassification.from_pretrained(
  "oliverguhr/german-sentiment-bert",
  { dtype: "fp32" },
); // downloads + caches ~437 MB once

const inputs = await tokenizer("Ich liebe diesen Ort 😍");
const { logits } = await model(inputs);
// softmax(logits) → { positive: 0.99, negative: 0.00, neutral: 0.01 }
// (config id2label: 0 positive · 1 negative · 2 neutral)

This page maps the labels to positiv / negativ / neutral, lightly cleans the text the way the model was trained (lowercase, strip digits/punctuation), and runs everything in a Web Worker so the UI never janks. "See inside" reuses the same loaded model to re-score occluded variants of the text and read off each word's pull.
