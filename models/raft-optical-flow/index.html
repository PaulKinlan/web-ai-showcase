RAFT — dense optical flow between two frames

Optical flow answers a deceptively hard question: where did every pixel go between two frames? RAFT (Recurrent All-Pairs Field Transforms) builds a 4D volume of all-pairs pixel correlations and iteratively refines a dense per-pixel motion field — a vector for every pixel giving the direction and speed it moved. It's the engine under video stabilisation, frame interpolation, motion segmentation, and tracking. Here it runs entirely on your device via ONNX Runtime Web: move a photo by a known amount and watch RAFT recover it, pixel by pixel. Nothing leaves the tab.

At a glance

Modelopencv/optical_flow_estimation_raft — RAFT (Teed & Deng, ECCV 2020), Sintel-trained
Taskoptical-flow (dense two-frame motion estimation) — run via raw ONNX Runtime Web (see note below)
Runtimeonnxruntime-web 1.21 (transformers.js has no optical-flow class / no optical-flow task)
Inputtwo frames [1, 3, 360, 480] float32 in 0–255 (plain RGB, fixed size) — frame 1 and frame 2
Outputflow field [1, 2, 360, 480] float32 — channel 0 = horizontal motion u, channel 1 = vertical motion v, in pixels
Quantizationfp32 ONNX — runs on the plain WASM execution provider, no WebGPU needed (the int8 export aborts on WASM)
BackendWebAssembly (single-threaded on GitHub Pages) — a 360×480 pass runs in ~6 s
Download~64 MB, cached after first load (auto-inits on return)
LicenseBSD-3-Clause (RAFT, princeton-vl) · MIT (ONNX conversion by PINTO0309, via OpenCV Zoo)
Web APIsWebAssembly, Web Workers, OffscreenCanvas, ImageBitmap — all Baseline widely available

Run it

Pick a photo and a motion. We build frame 2 by shifting the scene by a known number of pixels, so you can check RAFT's answer against the truth. Colour shows direction (see the wheel); brightness of colour shows speed. Grey = no motion.

Why raw ONNX Runtime Web?

Transformers.js registers no optical-flow model class and has no optical-flow pipeline task, so we run RAFT's ONNX graph directly with onnxruntime-web and hand-write the pre/post a pipeline would own: resize both frames to the model's fixed 360×480, pack them as [1,3,360,480] float32 in 0–255, and colour-code the returned [1,2,360,480] motion field. Everything — download, inference, rendering — happens on your device.
