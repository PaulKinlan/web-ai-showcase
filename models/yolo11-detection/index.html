YOLO11 — the newest YOLO, and the NMS is back

YOLO11 is Ultralytics' latest detector — more accurate than YOLOv10 at the same tiny size. But it makes the opposite design choice: where YOLOv10 trains an NMS-free head that hands back tidy, deduplicated boxes, YOLO11 keeps the classic dense head — one forward pass returns 8400 overlapping candidate boxes that you must clean up yourself with Non-Max-Suppression. Drop an image, then slide the IoU threshold and watch NMS collapse the duplicate flood in real time, entirely on your device.

See inside — the NMS step YOLOv10 removed

This is the whole point of YOLO11 vs YOLOv10. YOLO11's dense head emits a fixed [1, 84, 8400] tensor: 8400 candidate boxes, most of them near-duplicates of the same object. First we argmax the 80 class scores at each anchor to get its best label and confidence; then we drop everything below the score floor; then Non-Max-Suppression walks the survivors from most to least confident, throwing away any box that overlaps an already-kept box of the same class by more than the IoU threshold. Watch the numbers move as you drag the sliders — every step runs on the cached candidates, no re-inference.

Why it matters

YOLO11 is the current default when people say "just use YOLO" — it's the accuracy sweet spot at the nano size, and it runs at ~10 MB entirely in a page. Understanding that it needs NMS (and YOLOv10 doesn't) is the difference between a detector that works and one that draws three boxes around every object. Here you can feel both halves — the raw model output and the classical clean-up — with nothing uploaded.

  • Accuracy at the edge — YOLO11-nano beats YOLOv10-nano on COCO at a similar size, so an on-device detector is a little sharper for free.
  • Counting & inventory — tally people in a queue, cars in a lot, or items on a shelf from a photo, with the NMS knob tuned to your scene.
  • Live camera — one clean forward pass per frame plus a fast client-side NMS keeps a webcam detector responsive on modest hardware.
  • Teaching the pipeline — the IoU slider makes NMS legible: crank it up and duplicates reappear, crank it down and distinct objects get merged.

How the API works

Because transformers.js has no yolo11 class, we run the ONNX graph directly and hand-write the three steps a pipeline would normally own — letterbox in, decode the dense head out, and dedupe with NMS:

import * as ort from "onnxruntime-web";

const session = await ort.InferenceSession.create(modelBytes); // ~10 MB fp32 ONNX
// preprocess: resize longest edge → 640, pad to 640×640, /255 → Float32 [1,3,640,640]
const { output0 } = await session.run({ images: input });
// output0.dims === [1, 84, 8400]  (channel-major: value(c, a) = data[c*8400 + a])
// per anchor a: cx,cy,w,h = data[0..3][a];  classScores = data[4..83][a]
//   → argmax the 80 class scores, threshold, then run Non-Max-Suppression (IoU) to dedupe

This page runs the model once at a low floor (0.05) inside a Web Worker — letterboxing on an OffscreenCanvas and decoding the head there — then caches every candidate and re-runs the client-side NMS as you drag the sliders, so both knobs are instant and never re-run inference. The ONNX weights are fetched through Cache Storage, so a returning visit auto-initialises with no re-download.
