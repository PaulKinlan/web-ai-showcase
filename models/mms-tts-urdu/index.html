MMS-TTS Urdu — end-to-end Urdu speech, one network

This is the Urdu checkpoint of Meta's MMS-TTS family — a distinct model from the English MMS-TTS demo, the Hindi one and the Arabic one: its own native Perso-Arabic (Nastaliq) character vocabulary (58 symbols, is_uroman:false — native Urdu script written right-to-left, not romanised Latin) and its own Urdu-trained VITS weights. Urdu shares the Arabic script family but is a separate checkpoint with letters like ٹ/ڈ/ڑ/ں/ہ and its own pronunciation. VITS is a conditional VAE with normalizing flows and a stochastic duration predictor: one end-to-end network from text straight to a waveform, no separate vocoder, and the same Urdu sentence comes out with slightly different rhythm every time.

Why it matters

An Urdu voice that runs entirely on-device: no per-character bill, no text leaving the tab. Urdu is spoken by well over a hundred million people across Pakistan and India and the diaspora; VITS won't out-act a big expressive model, but it's the model you reach for when you need Urdu speech everywhere, offline, privately — with correct right-to-left rendering.

  • Offline Urdu read-aloud — narrate Urdu articles, forms or messages with no service to sign up for and nothing sent anywhere.
  • Accessibility — a built-in Urdu screen-voice for content, working without a network.
  • Embeddable Urdu voice — bundle an Urdu TTS inside an app or device UI, RTL-correct.
  • Natural variation — the stochastic duration predictor keeps repeated Urdu prompts from sounding mechanically identical.

How the API works

MMS-TTS is normally a one-liner with the text-to-speech pipeline. This ONNX export ships model.onnx at the repo root and omits a tokenizer.json, so instead of the high-level pipeline the worker loads the real VitsModel and reproduces the tokeniser exactly — safe here because this checkpoint is is_uroman:false, so its VitsTokenizer is a deterministic character tokeniser (no uroman romanisation): 

import { AutoModel, Tensor } from "@huggingface/transformers";

// vocab.json is byte-identical to canonical facebook/mms-tts-urd-script_arabic
const model = await AutoModel.from_pretrained(
  "naklitechie/mms-tts-ur-ONNX", { model_file_name: "model", subfolder: "", dtype: "fp32" });

// Faithful VitsTokenizer: lowercase, map each in-vocab char, interleave blank (id 0).
// VERIFIED byte-identical to transformers.js's real VitsTokenizer on a control language.
const ids = tokenizeUrdu("ایک ہی نیٹ ورک حروف کو براہِ راست آواز کی لہر میں بدل دیتا ہے۔");
const input_ids = new Tensor("int64", BigInt64Array.from(ids.map(BigInt)), [1, ids.length]);
const { waveform } = await model({ input_ids, attention_mask });
// waveform.data → Float32Array (mono), 16 kHz. Run again → slightly different (stochastic duration).

The manual tokeniser was checked byte-for-byte against transformers.js's real VitsTokenizer on a control language (Xenova/mms-tts-hin) — identical token ids — so the Urdu tokenisation is faithful, and the waveform is genuine VitsModel output. The Hindi demo uses the plain pipeline() because its export ships the full tokenizer files.
