Retrieve with mxbai, then rerank with a cross-encoder

This is how production search actually works. Stage one: mxbai-embed is a bi-encoder — it embeds the query and every document independently, so it's fast enough to scan a whole corpus and fetch a shortlist. Stage two: an MS MARCO cross-encoder reads the query and each shortlisted passage together, so it judges relevance more precisely — but is too slow to run over everything. Retrieve wide and cheap, then rerank narrow and sharp. Both on your device.

Stage 1 — mxbai shortlist (bi-encoder, cosine)

Stage 2 — cross-encoder rerank of that shortlist

retrieve –rerank –corpus –shortlist –

How the two connect

mxbai's independent query/document vectors make stage one a cheap cosine scan — great recall, approximate precision. The cross-encoder can't be precomputed (it needs the query and passage together), so you only run it on mxbai's top-k. The reranker's output order is the final ranking; watch it promote a passage mxbai ranked lower when the fine-grained query↔passage attention finds a better match. That's the recall-then- precision division of labour behind modern retrieval and RAG.
