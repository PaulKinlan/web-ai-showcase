mxbai-embed-large-v1 — a large, high-quality retriever

Mixedbread's flagship embedder turns a sentence into 1024 numbers — a big, expressive point in meaning space — and was trained with the AnglE objective to rank near the top of the MTEB retrieval benchmark. It brings two conventions worth knowing: a retrieval prompt you prepend to queries, and Matryoshka vectors you can safely truncate to trade quality for size. Type a few lines, search by meaning, and watch both levers move the results. Nothing leaves your device.

A/B — does the retrieval prompt help?

Embed the query below both ways — plain, and with the “Represent this sentence for searching relevant passages: ” prefix — and rank the sentences above with each. The prefix usually pushes the truly relevant lines up and the irrelevant ones down, widening the gap that makes retrieval work.

Matryoshka — quality vs size

mxbai's vector is trained so a leading slice is still a good embedding. Truncate 1024 → 512 → 256 → 128 → 64 (re-normalized) and see how much of the full-precision ranking survives at a fraction of the storage. This is how you cut a vector database's footprint without re-embedding.

Why it matters

Embeddings power search, recommendations, dedup, clustering and RAG. mxbai's pitch is quality: a large, MTEB-strong retriever you can still run in a tab, with two production levers baked in — the retrieval prompt that sharpens query/document matching, and Matryoshka truncation that shrinks your index on demand. And because it runs locally, the corpus and the queries never leave the device.

  • High-quality semantic search — rank a corpus by meaning with a top-tier retriever, client-side.
  • Prompted retrieval — the query prefix consistently improves ranking over embedding query and doc identically.
  • Right-sized vector stores — truncate to 512 or 256 dims to halve or quarter storage with graceful quality loss.
  • Retrieve-then-rerank — mxbai fetches a shortlist, a cross-encoder re-scores the top few for precision.

How the API works

Feature-extraction with mean pooling — and the one thing to remember is the query prompt:

import { pipeline } from "@huggingface/transformers";

const extract = await pipeline(
  "feature-extraction",
  "mixedbread-ai/mxbai-embed-large-v1",
); // downloads + caches ~337 MB once

const PROMPT = "Represent this sentence for searching relevant passages: ";

// Prompt the QUERY; embed documents plain:
const q = await extract([PROMPT + "how do I sign in?"], { pooling: "mean", normalize: true });
const d = await extract(["the login page rejects my password"], { pooling: "mean", normalize: true });

const sim = q.tolist()[0].reduce((s, x, i) => s + x * d.tolist()[0][i], 0); // cosine

// Matryoshka: keep the first N dims, then re-normalize.
function truncate(vec, n) { const v = vec.slice(0, n); const m = Math.hypot(...v); return v.map(x => x / m); }

This page asks the worker for the un-normalized pooled vectors so “See inside” can show the true magnitude and do Matryoshka truncation in JS. Inference runs in a Web Worker, so the matrix, projection, A/B and size sweep never jank the page.
