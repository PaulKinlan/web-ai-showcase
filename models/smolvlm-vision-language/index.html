SmolVLM — talk to a picture, on your device

SmolVLM is a tiny vision-language model: give it an image and a question, and it writes an answer. Describe a scene, read the text in a sign, count the cats, reason about what's happening — all streaming token by token on your GPU, with nothing uploaded.

Why it matters

A 256M-parameter model that understands images used to be unthinkable in a browser tab. SmolVLM makes visual understanding a client-side primitive — private, offline-capable, and free to run.

  • Automatic alt-text — describe every image on a page for screen-reader users, without sending photos to a server.
  • Screenshot Q&A — let users ask questions about a chart, form, or UI screenshot right in the app.
  • Private document triage — read and summarise a photographed receipt or note on-device, so sensitive images never leave.
  • Camera copilots — point-and-ask experiences ("what plant is this?", "what does this label say?") with no per-call cost.

How the API works

SmolVLM uses the vision-to-text model classes. You build a chat message whose content mixes an image placeholder and text, template it, run the processor, then generate — streaming tokens as they come: 

import { AutoProcessor, AutoModelForVision2Seq, load_image, TextStreamer }
  from "@huggingface/transformers";

const processor = await AutoProcessor.from_pretrained("HuggingFaceTB/SmolVLM-256M-Instruct");
const model = await AutoModelForVision2Seq.from_pretrained(
  "HuggingFaceTB/SmolVLM-256M-Instruct",
  { dtype: "q4f16", device: "webgpu" },   // WebGPU required
);

const image = await load_image(imageURL);
const messages = [{ role: "user", content: [{ type: "image" }, { type: "text", text: prompt }] }];
const text = processor.apply_chat_template(messages, { add_generation_prompt: true });
const inputs = await processor(text, [image]);

const streamer = new TextStreamer(processor.tokenizer,
  { skip_prompt: true, skip_special_tokens: true, callback_function: (t) => append(t) });
await model.generate({ ...inputs, max_new_tokens: 150, streamer });

This page runs exactly that, in a Web Worker, and gates on a real WebGPU adapter first — if none exists it shows a labelled needs-WebGPU state instead of a fake answer.
