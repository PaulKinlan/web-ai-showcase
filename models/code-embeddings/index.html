Jina Code Embeddings — search code by meaning, not keywords

Text embedders map sentences into meaning-space. This one does it for code: Jina's v2 base-code model was contrastively trained on 150M+ pairs across 30 programming languages, so an English description — "read a file and parse JSON" — lands right next to the function that implements it, and two functions that do the same job in Python and Go land next to each other even though they share almost no tokens. Type a query, rank a bank of snippets by meaning, and watch keyword search fall apart where semantic search holds up. Nothing leaves your device.

See inside — the vector, and NL↔code alignment

The encoder emits one 768-number vector per token; mean-pooling (mask-aware) then L2-normalizing collapses that to a single unit vector per snippet, so cosine similarity is a plain dot product. The magic of a code embedder is alignment: the English query and the function that implements it are trained to point the same way. Why it matters

Grep finds the tokens you typed; a code embedder finds the code you meant. That difference unlocks search and tooling that used to need a hosted service — now it runs in a tab, over a private repo, with nothing uploaded.

  • Natural-language code search — "where do we validate the JWT?" returns the right function even if it never says "JWT".
  • Duplicate & near-duplicate detection — collapse copy-pasted helpers that drifted apart, entirely client-side.
  • Cross-language clone finding — spot the same algorithm reimplemented in Python and Go during a migration.
  • Retrieval for a coding assistant — fetch the most relevant snippets to feed a local LLM, on-device (see the multi-model demo).

How the API works

Feature-extraction with mean pooling — and query and code are embedded identically (no prefix):

import { pipeline } from "@huggingface/transformers";

const extract = await pipeline(
  "feature-extraction",
  "jinaai/jina-embeddings-v2-base-code",
  { dtype: "q8" },
); // downloads + caches ~162 MB once

// Embed an English query and a code snippet the SAME way:
const out = await extract(
  ["reverse a string", "def reverse(s):\n    return s[::-1]"],
  { pooling: "mean", normalize: true },
);
const [q, code] = out.tolist();

// unit-length vectors → cosine similarity is just the dot product:
const sim = q.reduce((s, x, i) => s + x * code[i], 0); // high — the query describes the code

This page asks the worker for the un-normalized pooled vectors so "See inside" can show the real pre-normalization magnitude, then normalizes them in JS. Inference runs in a Web Worker, so ranking and the matrix never jank the page.
