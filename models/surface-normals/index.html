Surface normals — which way does every pixel face?

Depth tells you how far a pixel is; a surface normal tells you which way it faces — the little arrow sticking out of the surface in 3D. Sapiens predicts one normal per pixel from a single photo, giving you the classic pastel normal map that graphics engines use to relight, emboss, and fake geometry. It runs entirely on your device.

Heavy model, honest speed. This is a ~444 MB int8 ViT. On WebGPU it would be seconds, but this demo ships the universal CPU (WASM) path — so the first estimate takes roughly 60–90 s in single-threaded WebAssembly. That is real, on-device inference, not a canned result. Once the normal map is computed, relighting and inspection are instant (they reuse the cached normals — no re-run).

Why it matters

A per-pixel normal map is the single most useful ingredient for making a flat photo behave like a 3D surface — no mesh, no scan, no special hardware. Once you know which way every pixel faces, client-side effects open up that depth alone can't do:

  • Relighting — move a virtual light and the face lights up correctly, because shading is just the angle between the surface normal and the light (N·L).
  • AR material & makeup try-on — normals tell you how light should bounce, so overlays sit on the surface instead of floating.
  • Emboss / bump / detail enhancement — drive a bump-map look or sharpen surface detail directly from geometry.
  • Depth + normals = 2.5D — combine "how far" with "which way" for occlusion-aware compositing and richer relighting.

Use cases

Basics

One photo in, a colour-encoded surface-normal map out — read orientation as colour.

Practical

Relighting studio — drag a light around and watch the face shade correctly from its normals (an AR-lighting concept).

Wild

Three-point light rig — key, fill and rim lights in colour, all driven by the normal field.

Multi-model

Normals + depth → 2.5D relight: shade by orientation, deepen by distance.

How the API works

transformers.js has no sapiens class, so we run the ONNX graph directly. It's a clean image→image model — normalize in, unit-vector field out:

import * as ort from "onnxruntime-web";

const session = await ort.InferenceSession.create(sapiensBytes, { executionProviders: ["wasm"] });

// image → [1,3,1024,768], ImageNet-normalized
const input = new ort.Tensor("float32", preprocess(image), [1, 3, 1024, 768]);
const { predicted_normal } = await session.run({ pixel_values: input });
// predicted_normal → [1,3,512,384] raw XYZ vectors; L2-normalize per pixel → unit surface normals
// RGB-encode with (n * 0.5 + 0.5) * 255 to get the classic pastel normal map

Inference runs in a Web Worker, and the RGB normal-map composite is built there too (OffscreenCanvas → transferred ImageBitmap), so the ~444 MB model never freezes the page. The unit-normal field is transferred back so relighting and pixel-inspection run without re-inference.
