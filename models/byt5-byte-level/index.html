ByT5-small — no tokenizer, just bytes

Almost every language model chops text into sub-word tokens from a fixed vocabulary — and that vocabulary decides what the model can even see. ByT5 throws the vocabulary away. It reads raw UTF-8 bytes: every character becomes one to four bytes, and each byte is a token. That one change makes it work identically across every language, script, emoji, and mangled input — nothing is ever "out-of-vocabulary." Type anything below and watch it become bytes, then watch the model fill in a blank — all locally, nothing uploaded.

This is the small (~300M) base checkpoint — pre-trained on byte-span denoising, not instruction-tuned. So calibrate expectations: it is genuinely good at showing the byte mechanism and does rough fill-in-the-blank completions, but it is not a chat model and its guesses are often loose. The page shows you the real output, right or wrong — never a canned answer.

See inside — text is just bytes (no vocabulary)

This is the whole tokenizer, live. As you type, each character is UTF-8 encoded to one or more bytes, and each byte becomes a token id (byte + 3). There is no vocabulary file, no merges, no [UNK] — a Latin letter, a Kanji, an emoji, and a math symbol all reduce to bytes the same way. Why it matters

A model with no vocabulary is a different kind of tool. It can't be tripped up by a script it wasn't trained to tokenize, a novel emoji, a typo, or deliberately corrupted text — everything is bytes, so everything is in-domain. That robustness is why byte-level models are used where input is messy or multilingual.

  • Truly multilingual by construction — one model, every script and writing system, with no per-language vocabulary to ship or blow up your token budget.
  • Noise & typo robustness — fine-tuned ByT5 models excel at spelling correction and normalization precisely because a corrupted byte is still just a byte, never an unknown token.
  • Nothing is out-of-vocabulary — emoji, rare Unicode, code, mixed scripts, or a user smashing the keyboard all encode losslessly. No [UNK], ever.
  • Character-level tasks — transliteration, grapheme-to-phoneme, and edit-level work are natural when the model sees the actual characters, not sub-word chunks.

How the API works

Because ByT5 is token-free, we skip the tokenizer and feed byte ids straight to the real ONNX model:

import { T5ForConditionalGeneration, Tensor } from "@huggingface/transformers";

const model = await T5ForConditionalGeneration.from_pretrained(
  "onnx-community/byt5-small-ONNX", { device: "wasm", dtype: "q8" },
); // downloads + caches ~290 MB once

// The ENTIRE tokenizer: UTF-8 encode, then add 3. No vocabulary, no [UNK].
const ids = [...new TextEncoder().encode("Roses are red, violets are ")]
  .map((b) => b + 3);
ids.push(258); // <extra_id_0> — the span to fill
ids.push(1);   // </s>

const input_ids = new Tensor("int64", BigInt64Array.from(ids.map(BigInt)), [1, ids.length]);
const attention_mask = new Tensor("int64", BigInt64Array.from(ids.map(() => 1n)), [1, ids.length]);
const out = await model.generate({ input_ids, attention_mask, max_new_tokens: 40 });

// Decode: subtract 3, back to bytes, back to text.
const bytes = [...out.data].map(Number).filter((id) => id >= 3 && id <= 257).map((id) => id - 3);
new TextDecoder().decode(new Uint8Array(bytes)); // → the filled text

This page runs exactly that in a Web Worker so the UI never janks, and the byte mapping is the single source of truth shared with the "see inside" view above.
