A tiny model has a ceiling. Here are increasingly nasty challenges — three-table joins, nested subqueries, conditional aggregation, "top-N per group". Run each and judge the SQL yourself: which does it nail, and where does it hallucinate a column or drop a clause? Editing the schema or question is the whole point — probe it.

What to watch for

Common tiny-model failure modes: inventing a column that isn't in the schema, joining on the wrong key, flattening a "per group" question into a single aggregate, or truncating a long query. When it fails, the failure is honest and on the page — that is far more instructive than a model that always looks right. For production, treat this as a first draft and validate.
