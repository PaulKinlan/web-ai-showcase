Qwen3 1.7B — a hybrid thinking model on your GPU

Qwen3 is the first model in this catalogue that can think out loud on demand. It's a single 1.7B model with two modes: flip thinking on and it emits a <think>…</think> chain of reasoning before answering — you watch the trace stream live — or flip it off for a fast, direct reply. One model, two behaviours, toggled with Qwen3's /think and /no_think switches. It runs entirely in your browser on MLC's WebLLM engine: streaming tokens, sampling controls, a real system prompt, nothing uploaded and no API bill. WebGPU only — a ~1.1 GB download, cached after first load.

How it differs from the other chat pages: Qwen3 is a hybrid reasoning model — the only one here that exposes an explicit, toggleable chain-of-thought — and a distinct Qwen architecture (Apache-2.0) from Alibaba, separate from the Gemma-family (Gemma 3 1B), Mistral (Mistral 7B) and Llama-family models. It's also newer than the Qwen2.5 0.5B WASM page: a bigger, reasoning-capable successor that runs on the GPU.

Why it matters

Reasoning models were, until recently, a server-side, premium-API affair. Qwen3 on WebLLM puts a genuinely capable hybrid reasoner on the client: for easy prompts you turn thinking off and get a fast reply; for hard ones — multi-step maths, planning, careful extraction — you turn it on and the model works through the problem in a visible trace before committing to an answer. All of it private by construction, free to run, and available offline once cached.

  • Show your work — the reasoning trace is a first-class UI surface: audit how the model reached an answer, or hide it for a clean reply. No other model here exposes this.
  • Spend compute only when it helps — a per-request thinking toggle lets you pay the reasoning cost on the 10% of prompts that need it and stay fast on the rest.
  • Private step-by-step assistants — tutoring, debugging, and "explain your reasoning" flows that never leave the device.
  • On-device RAG with reasoning — pair it with a local embedding model for grounded answers that can reason over the retrieved passages (see the multi-model demo).

How the API works

WebLLM gives you an OpenAI-shaped chat API backed by a WebGPU engine. You create the engine (which downloads and compiles the weights), then stream a completion. Qwen3 has a real system role, so no folding is needed; the hybrid mode is selected by appending /think or /no_think to the last user turn. In thinking mode the model streams a <think>…</think> segment (or, on some builds, a separate reasoning_content delta) before the answer — this page separates the two streams in worker.js: 

import { webGPUAdapterAvailable, createEngine }
  from "/web-ai-showcase/lib/webllm.js";

// WebLLM is WebGPU-only — gate before doing anything (headless requestAdapter() → null).
if (!(await webGPUAdapterAvailable())) {
  showNeedsWebGPU();          // never a faked reply
} else {
  const engine = await createEngine({
    model: "Qwen3-1.7B-q4f16_1-MLC",
    onProgress: (p) => setProgress(p.text, p.progress),   // ~1.1 GB, cached after first load
  });

  const thinking = true;      // per-request toggle
  const messages = [
    { role: "system", content: "You are a concise assistant." },
    { role: "user", content: "Is 97 prime? " + (thinking ? "/think" : "/no_think") },
  ];

  const stream = await engine.chat.completions.create({
    messages, temperature: 0.7, top_p: 0.8, max_tokens: 512, stream: true,
  });
  for await (const chunk of stream) {
    const d = chunk.choices[0].delta;
    if (d.reasoning_content) appendReasoning(d.reasoning_content); // the <think> trace
    if (d.content)           appendAnswer(d.content);              // the final answer
  }
  // engine.interruptGenerate() stops decoding early; engine.runtimeStatsText() reports tok/s.
}

Everything here runs exactly that path in worker.js, and gates on a real WebGPU adapter first. No adapter → a labelled needs-WebGPU state, never a canned answer.
