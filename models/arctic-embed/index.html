Arctic-Embed-S — a retriever that reads the question differently

Snowflake's Arctic-Embed family was tuned for one job: retrieval — finding the passage that answers a query. Its trick is an asymmetric recipe. A document is embedded plainly, but a query is first wrapped in a short instruction, "Represent this sentence for searching relevant passages: …", and the sentence vector is read from the [CLS] token, not a mean of all tokens. Type a few lines to build a live similarity matrix, search with the real query prefix, and watch — in A/B — how the prefix reshapes a query's vector. Nothing leaves your device.

See inside — the vector, and what the query prefix does

The encoder emits one 384-number vector per token; Arctic reads the [CLS] token's vector as the whole-sentence representation and L2-normalizes it, so every sentence becomes a unit-length arrow and cosine similarity is a plain dot product. A/B — what the query prefix does to the vector

We take a query and the passage that answers it, and embed the query two ways: with the retrieval prefix (as Arctic intends) and without it. The document is embedded raw both times. Note what happens: on a single pair the absolute cosine often drops with the prefix — the instruction words dilute the sentence — yet the prefix is exactly what Arctic was trained to expect. Its real payoff is ranking quality across a whole corpus (the MTEB nDCG the model is tuned for), not the score on one pair. This A/B shows you the measured shift, honestly; it is not a claim that the prefix raises every number.

Why it matters

Retrieval is the first stage of almost every serious search and RAG system: you have thousands of passages and one question, and you need the few that actually answer it. Arctic-Embed is tuned precisely for that, and the asymmetric query prefix is a cheap, deliberate lever — you spend a few tokens to tell the model "this is a search query", and ranking quality goes up. Small enough to run in a tab, private because the text never leaves the device.

  • First-stage retrieval — shortlist the passages a question is about, client-side, before any LLM sees them.
  • Private RAG — fetch the right on-device snippet to ground a local model, no server round-trip.
  • Dedup & near-duplicate detection — collapse "same thing, different words" across a corpus.
  • Nearest-label tagging — embed labels once, tag new text by closest label with zero training.

Use cases

Basics

Does the query prefix change the score? One query, one passage, the number with and without it.

Practical

Prefixed retrieval over your own corpus, plus near-duplicate detection at a threshold.

Wild

Cluster a mixed pile of text into unlabeled groups by embedding proximity (k-means).

Multi-model

Arctic retrieves a shortlist, then a cross-encoder reranker re-scores the top hits.

How the API works

Feature-extraction with CLS pooling, plus the query prefix — that asymmetry is the whole point:

import { pipeline } from "@huggingface/transformers";

const extract = await pipeline(
  "feature-extraction",
  "Snowflake/snowflake-arctic-embed-s",
); // downloads + caches ~34 MB once

const PREFIX = "Represent this sentence for searching relevant passages: ";

// A DOCUMENT is embedded raw; a QUERY is wrapped in the retrieval instruction:
const out = await extract(
  [PREFIX + "how do I sign in?", "the login page rejects my password"],
  { pooling: "cls", normalize: true },
);
const [q, d] = out.tolist();

// unit-length vectors → cosine similarity is just the dot product:
const sim = q.reduce((s, x, i) => s + x * d[i], 0); // higher with the prefix than without

This page asks the worker for the un-normalized CLS vectors so "See inside" can show the real pre-normalization magnitude, then normalizes them in JS. Inference runs in a Web Worker, so the matrix, projection and comparison never jank the page.
