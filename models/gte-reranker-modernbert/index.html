GTE-reranker-ModernBERT — rank passages by relevance, up to 8192 tokens

Alibaba's GTE reranker built on the ModernBERT backbone — the 2025 encoder redesign (rotary embeddings, GeGLU, alternating local/global attention) with an 8192-token context. Instead of turning a query and a passage into separate vectors and comparing them, a cross-encoder reads both together and outputs a single relevance score — and because ModernBERT handles long inputs, it reranks whole documents a base-BERT reranker would have to truncate. Give it a query and candidates and watch it re-sort them; then look inside at the raw logit per pair. Nothing leaves your device.

Why it matters

Rerankers are the precision stage of modern search. A fast retriever (a keyword index or embeddings) casts a wide net and returns, say, the top 50 candidates; a cross-encoder then reads each candidate against the query and reorders them with far higher accuracy. GTE-reranker-ModernBERT brings two things to that stage: the 8192-token context of ModernBERT, so it can score long passages and whole documents without chopping them to 512 tokens, and ModernBERT's efficient attention, so it stays light enough to run in a tab. The precise, expensive step happens on-device with nothing sent anywhere.

  • Long-document RAG — rerank full sections or pages, not just truncated snippets, so the LLM reads the chunk that actually answers the question.
  • Two-stage search — bolt onto any first-stage retriever (BM25 or embeddings) as a drop-in precision filter over the shortlist.
  • Answer selection — pick the best reply from candidates even when they're phrased very differently from the question, and resist keyword-stuffed decoys.
  • Same-backbone pipeline — pair it with the GTE-ModernBERT bi-encoder embedder (see the multi-model demo) for a retrieve-then-rerank stack sharing one architecture.

Use cases

Basics

Rank a handful of passages against a query by cross-encoder relevance.

Practical

Two-stage RAG: retrieve a shortlist, then rerank it with ModernBERT for precision.

Wild

Keyword-stuffed hard negatives: decoys packed with the query's words that answer the wrong thing.

Multi-model

GTE-ModernBERT embeddings retrieve, GTE-reranker-ModernBERT reorders — same backbone, two stages.

Contrast with the multilingual BGE-reranker-v2-m3: same cross-encoder idea, but this one uses the ModernBERT backbone for an 8192-token context (long-document reranking) where v2-m3 trades context for 100+ language coverage.

How the API works

A cross-encoder scores query–passage pairs; we read the raw logit off the model directly:

import { AutoTokenizer, AutoModelForSequenceClassification } from "@huggingface/transformers";

const model_id = "Alibaba-NLP/gte-reranker-modernbert-base";
const tokenizer = await AutoTokenizer.from_pretrained(model_id);
const model = await AutoModelForSequenceClassification.from_pretrained(model_id, { dtype: "q8" });

// Encode the query with each passage AS A PAIR — [CLS] query [SEP] passage [SEP]:
const inputs = tokenizer(
  passages.map(() => query),
  { text_pair: passages, padding: true, truncation: true },
);
const { logits } = await model(inputs); // one relevance logit per pair — higher = more relevant
// sort the passages by logits.data — that's your reranking. ModernBERT's 8192-token context means
// each passage can be a long document, not a truncated snippet.

This page loads the model through the shared loader as a text-classification pipeline, then reuses pipe.model + pipe.tokenizer to read the raw logits (the pipeline would squash them through a sigmoid). All inference runs in a Web Worker so ranking never janks the page.
