Multi-model: retrieve, then rerank

This is how real search pipelines are built. Stage 1: all-mpnet-base-v2 (a bi-encoder) embeds every passage once and finds a shortlist by cosine — fast, scales to millions of documents. Stage 2: MS MARCO MiniLM (a cross-encoder) rereads the query and each shortlisted passage together, letting attention flow across both, and rescores them — slow per pair, but far more precise. Two models, two workers, both local.

Why two stages? The bi-encoder never sees the query and passage together, so near-misses ("battery lasts ten hours" vs "charging takes ninety minutes") stay uncomfortably close in cosine. The cross-encoder reads each pair jointly and separates them by a wide logit margin — but it must run once per pair, so you only feed it the shortlist. ↑/↓ badges show what the rerank changed. And try the third chip for honesty's sake: MS MARCO was trained on web search, and on a niche corpus the reranker can be confidently wrong too — stage 2 is a precision tool, not an oracle.
