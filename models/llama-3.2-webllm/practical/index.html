Practical: a local tool router

The building block under every "agent": given a plain-English request and a menu of tools, the model must pick one tool and fill in its arguments as strict JSON — the step your code then dispatches to a real function. It leans on instruction-following and schema-adherence, exactly where the 3B pulls ahead of the 1B. WebLLM's JSON mode guarantees the output parses; the whole decision runs on your GPU with no server.

Available tools (the model must choose exactly one):

  • get_weather(city, unit?) — current conditions
  • set_reminder(text, datetime) — schedule a reminder
  • send_email(to, subject, body) — draft an email
  • convert_currency(amount, from, to) — FX conversion
  • search_web(query) — look something up

How this stays reliable

We describe the tool schemas in the system prompt and pass response_format: { type: "json_object" } to WebLLM, which constrains decoding so the tokens can only form valid JSON — no "sometimes it wraps the call in prose" failure. The model returns { "tool": "…", "arguments": { … } }; the page parses it, checks the tool name against the known list, and shows the dispatch your code would perform. A wrong or unknown tool is flagged rather than hidden — this is a router you could actually wire to real functions.
