LFM2 — Liquid AI's hybrid on-device LLM

Most small browser LLMs (SmolLM, Qwen, Llama, Gemma) are scaled-down transformers. LFM2 is a different shape: Liquid AI's second-generation foundation model (2025) interleaves short-range gated convolution blocks with grouped-query attention — a hybrid designed from the ground up for fast CPU/edge inference, not a shrunk GPT. This is the 350M instruct model, running locally via Transformers.js: it streams real tokens on WebGPU (q4f16) when you have a GPU, and on plain CPU/WASM (fp32) when you don't. Nothing leaves your device.

At a glance

ModelLiquidAI/LFM2-350M · run via the onnx-community/LFM2-350M-ONNX Transformers.js build
ArchitectureHybrid — gated short convolution blocks + grouped-query attention (the lfm2 model class), ~354M params. Built for edge/CPU latency.
Task / pipelinetext-generation (instruct chat, applies LFM2's chat template)
RuntimeTransformers.js 4.2.0, pinned locally in this model's worker (the lfm2 class isn't in the shared 3.7.5 pin — isolated escape hatch, same as SAM2).
Backend & dtypeWebGPU → q4f16 (~260 MB, fast) · CPU/WASM → fp32 (~1.4 GB, works with no GPU). The page probes for a real GPU adapter and picks; both are real.
Why two dtypesThe quantized ONNX block-quantizes the token-embedding gather (GatherBlockQuantized), a WebGPU-only ORT-Web kernel — so on CPU we load the fp32 export, which uses a plain gather.
Context length32,768 tokens
LicenseLFM Open License v1.0 (review before production use)
Web APIsWebAssembly, Web Workers (Baseline); WebGPU for the fast path (progressive enhancement)

Why it matters

      

        LFM2's pitch is latency and footprint on the device: the hybrid conv+attention
        design targets fast decoding on CPUs and small accelerators, which is exactly the browser's
        constraint. A 350M instruct model that runs — really runs — with no GPU means private, offline
        assistants on ordinary hardware.
      

      
    
        
  • Private local chat — a small assistant that never sends a token to a server.

How the API works

      
Standard Transformers.js text-generation — the LFM2 specifics are the version pin and the device-adaptive dtype:

      
// LFM2's model class lands in Transformers.js 4.x, so pin it locally in the worker:
import { pipeline, TextStreamer } from "@huggingface/transformers"; // @4.2.0

// WebGPU → q4f16 (fast); no GPU → fp32 (the WASM EP can't run the block-quantized gather):
const gpu = "gpu" in navigator && await navigator.gpu.requestAdapter();
const generator = await pipeline("text-generation", "onnx-community/LFM2-350M-ONNX", {
  device: gpu ? "webgpu" : "wasm",
  dtype:  gpu ? "q4f16"  : "fp32",
});

const messages = [{ role: "user", content: "Explain why the sky is blue in two sentences." }];
const streamer = new TextStreamer(generator.tokenizer, { skip_prompt: true, skip_special_tokens: true,
  callback_function: (t) => append(t) });                 // stream tokens as they arrive
await generator(messages, { max_new_tokens: 256, do_sample: true, temperature: 0.3, streamer });

      

        Inference runs in a Web Worker so token streaming never blocks the page. Verified on this stack
        (headless Chrome, no GPU, fp32/WASM): greedy decode produced
        "The sky appears blue because of a phenomenon called Rayleigh scattering, where shorter
        wavelengths of light, like blue, scatter more" — real, coherent, non-degenerate.
