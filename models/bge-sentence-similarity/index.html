bge-small-en-v1.5 — rank text by meaning, retrieval-grade

A stronger retrieval embedder than MiniLM: BAAI's BGE turns each sentence into a 384-number vector tuned for search and similarity. Give it a query and a handful of candidates and it ranks them by cosine similarity — and it knows a trick MiniLM doesn't: a query instruction that sharpens short-query-to-passage search. Everything runs locally; nothing leaves your device.

Why it matters

Embeddings are the workhorse behind search, recommendations, dedup, clustering, and RAG. BGE is a retrieval-tuned embedder — on the MTEB benchmark it out-ranks general models like MiniLM for finding the right passage — yet at 34 MB it still runs in a tab. So you can ship meaning-based search with no vector database, no API bill, and no user text leaving the device.

  • Semantic search — rank a corpus by meaning, beating keyword search on synonyms and paraphrases, entirely client-side.
  • Deduplication & clustering — collapse "same thing, different words" tickets, notes, or reviews with no labels.
  • Retrieve-then-rerank — BGE fetches a shortlist fast; a cross-encoder reranker then reorders it precisely.
  • Private RAG — pull the right snippet to feed a local LLM, all on-device.

How the API works

Feature-extraction with CLS pooling is the whole recipe — plus the query instruction for asymmetric search:

import { pipeline } from "@huggingface/transformers";

const extract = await pipeline(
  "feature-extraction",
  "Xenova/bge-small-en-v1.5",
); // downloads + caches ~34 MB once

// BGE v1.5 is trained with CLS pooling (NOT mean pooling):
const embed = (texts) => extract(texts, { pooling: "cls", normalize: true });

// Asymmetric search: prefix the QUERY only, leave passages bare.
const INSTR = "Represent this sentence for searching relevant passages:";
const q = await embed([`${INSTR} how to keep a battery healthy`]);
const passages = await embed([
  "Keep lithium-ion charge between 20 and 80 percent.",
  "The best hiking trails in the Lake District.",
]);

// unit vectors → cosine similarity is just the dot product:
const score = (a, b) => a.reduce((s, x, i) => s + x * b[i], 0);
const q0 = q.tolist()[0], ps = passages.tolist();
ps.map((p) => score(q0, p)); // → [0.71, 0.28] — battery line wins

This page asks the worker for the un-normalized CLS vectors so "See inside" can show the real pre-normalization magnitude, then normalizes in JS. Inference runs in a Web Worker so ranking never janks the page. Use the instruction for short-query search; for symmetric similarity (STS, dedup, paraphrase) prefix nothing on either side.
