DialoGPT — the Reddit-trained chatbot that runs on plain WASM

DialoGPT is GPT-2 fine-tuned on 147M multi-turn conversations scraped from Reddit. It has one clever trick for holding a conversation: the entire dialogue so far is fed back as a single sequence, with each turn separated by the model's end-of-text token (<|endoftext|>). The model just continues that sequence — and whatever it writes up to the next end-of-text token is its reply. It's old (2019) and can be terse or odd, but it's a wonderfully legible window into how a chatbot is really just a text completer with a turn separator.

Heads up: DialoGPT is a 2019 research model trained on unfiltered Reddit. Replies can be short, repetitive, off-topic, or blunt — that's the model being itself, not a bug. It's here as a legible teaching example of the conversational mechanism, not as a modern assistant.

Run it — chat

Send a message and DialoGPT replies, streaming token by token from a Web Worker. Every turn you both say is kept and re-fed as the EOS-joined context, so it (loosely) remembers the conversation. Switch between greedy (always the single most likely token — safe, terse) and sampling (temperature + top-k — livelier, riskier).

See inside — the EOS-joined dialogue history

This is the exact string DialoGPT sees. Every turn — yours and its — is concatenated into one sequence, with the <|endoftext|> token marking each boundary. There is no separate "memory"; the whole conversation is just re-tokenised and re-fed each time, and the model completes it. When the model emits its own end-of-text token, the reply is done.

Why it matters

DialoGPT makes the anatomy of a chatbot obvious: there is no state machine and no memory store — just a text completer and a turn-separator token. Every modern chat model, from tiny on-device assistants to the largest hosted ones, still works this way underneath (with fancier turn formats and alignment on top). Seeing it in a 124M model that runs on plain WASM demystifies the whole idea.

  • Teach how chatbots actually work — the EOS-joined transcript, live, with nothing hidden.
  • Offline, private conversational filler — placeholder dialogue, NPC chatter, or test fixtures with nothing leaving the tab.
  • A quick-reply generator — draft a couple of candidate responses to a message, then pick or edit (see Practical).
  • A stage in a pipeline — generate a reply, then run a second model to read its tone or emotion (see Multi-model).

Use cases

Basics

A minimal chat: one message in, one streamed reply out, with the context tokens shown.

Multi-model

DialoGPT replies, then an emotion model reads the mood of what it just said.

How the API works

The turn format is the whole idea — join the dialogue with the end-of-text token and let the model continue:

import { AutoTokenizer, AutoModelForCausalLM, Tensor } from "@huggingface/transformers";

const tok = await AutoTokenizer.from_pretrained("onnx-community/DialoGPT-small-ONNX");
const model = await AutoModelForCausalLM.from_pretrained("onnx-community/DialoGPT-small-ONNX", { dtype: "q8" });

const EOS = tok.decode([model.config.eos_token_id]); // "<|endoftext|>"
// The conversation so far, each turn separated (and terminated) by EOS:
const context = ["Cats or dogs?", "Dogs, definitely.", "Why dogs?"].join(EOS) + EOS;

let ids = Array.from(tok(context).input_ids.data, Number);
// then a greedy/sampling forward-pass loop over model(ids) until it emits EOS — that span is the reply.

We run the decode loop ourselves with AutoModelForCausalLM (rather than the text-generation pipeline) for two reasons: this single-file ONNX export doesn't drive transformers.js's generate() KV-cache path, and running the loop by hand lets us stream every token and show the real next-token distribution. It all runs in a Web Worker so the chat stays responsive.
