MMS-TTS Odia — end-to-end Odia speech, one network

This is the Odia (Oriya) checkpoint of Meta's MMS-TTS family — a distinct model from the Hindi, Tamil and Malayalam demos: its own native Odia-script character vocabulary (76 symbols, is_uroman:false — native Odia, not romanised Latin, and the distinctive rounded script of Odisha) and its own Odia-trained VITS weights, so it speaks Odia orthography — vowels ଅ/ଆ/ଇ, consonants କ/ଙ/ଚ, conjuncts and vowel signs — that the other checkpoints cannot. VITS is a conditional VAE with normalizing flows and a stochastic duration predictor: one end-to-end network from text straight to a waveform, no separate vocoder, and the same Odia sentence comes out with slightly different rhythm every time.

Why it matters

An Odia voice that runs entirely on-device: no per-character bill, no text leaving the tab. Odia is a classical language spoken by tens of millions across Odisha and the diaspora; VITS won't out-act a big expressive model, but it's the model you reach for when you need Odia speech everywhere, offline, privately.

  • Offline Odia read-aloud — narrate Odia articles, forms or messages with no service to sign up for and nothing sent anywhere.
  • Accessibility — a built-in Odia screen-voice for content, working without a network.
  • Embeddable Odia voice — bundle an Odia TTS inside an app or device UI.
  • Natural variation — the stochastic duration predictor keeps repeated Odia prompts from sounding mechanically identical.

How the API works

MMS-TTS is normally a one-liner with the text-to-speech pipeline. This ONNX export ships model.onnx at the repo root and omits a tokenizer.json, so instead of the high-level pipeline the worker loads the real VitsModel and reproduces the tokeniser exactly — safe here because this checkpoint is is_uroman:false, so its VitsTokenizer is a deterministic character tokeniser (no uroman romanisation): 

import { AutoModel, Tensor } from "@huggingface/transformers";

// vocab.json is byte-identical to canonical facebook/mms-tts-ory
const model = await AutoModel.from_pretrained(
  "naklitechie/mms-tts-or-ONNX", { model_file_name: "model", subfolder: "", dtype: "fp32" });

// Faithful VitsTokenizer: lowercase, map each in-vocab char, interleave blank (id 0).
// VERIFIED byte-identical to transformers.js's real VitsTokenizer on a control language.
const ids = tokenizeOdia("ଗୋଟିଏ ନେଟୱାର୍କ ଅକ୍ଷରକୁ ସିଧାସଳଖ ଧ୍ୱନି ତରଙ୍ଗରେ ପରିଣତ କରେ।");
const input_ids = new Tensor("int64", BigInt64Array.from(ids.map(BigInt)), [1, ids.length]);
const { waveform } = await model({ input_ids, attention_mask });
// waveform.data → Float32Array (mono), 16 kHz. Run again → slightly different (stochastic duration).

The manual tokeniser was checked byte-for-byte against transformers.js's real VitsTokenizer on a control language (Xenova/mms-tts-hin) — identical token ids — so the Odia tokenisation is faithful, and the waveform is genuine VitsModel output. The Hindi demo uses the plain pipeline() because its export ships the full tokenizer files.
