MMS-TTS Hindi — end-to-end Hindi speech, one network

This is the Hindi checkpoint of Meta's MMS-TTS family — a distinct model from the English MMS-TTS demo: its own Devanagari character vocabulary (72 symbols, is_uroman:false — native Devanagari, not romanised Latin) and its own Hindi-trained VITS weights, so it speaks Hindi orthography — consonants क/ख/ग, matras ि/ी/ु/ू, anusvara ं, chandrabindu ँ — that the English model cannot. VITS is a conditional VAE with normalizing flows and a stochastic duration predictor: one end-to-end network from text straight to a waveform, no separate vocoder, and the same Hindi sentence comes out with slightly different rhythm every time.

Why it matters

A Hindi voice that runs entirely on-device: no per-character bill, no text leaving the tab, and a tiny ~38 MB footprint you can ship inside a PWA or kiosk. Hindi is a first language for hundreds of millions of people; VITS won't out-act a big expressive model, but it's the model you reach for when you need Hindi speech everywhere, offline, privately.

  • Offline Hindi read-aloud — narrate Hindi articles, forms or messages with no service to sign up for and nothing sent anywhere.
  • Accessibility — a built-in Hindi screen-voice for content, working without a network.
  • Tiny, embeddable Hindi voice — ~38 MB makes it practical inside an app or device UI.
  • Natural variation — the stochastic duration predictor keeps repeated Hindi prompts from sounding mechanically identical.

How the API works

MMS-TTS Hindi is a one-liner with the text-to-speech pipeline — the Hindi checkpoint is picked by its id:

import { pipeline } from "@huggingface/transformers";

const tts = await pipeline("text-to-speech", "Xenova/mms-tts-hin", { dtype: "q8" });

const out = await tts("एक ही नेटवर्क अक्षरों को सीधे ध्वनि तरंग में बदल देता है।");
// out.audio         → Float32Array waveform (mono)
// out.sampling_rate → 16000
// Run it again with the same text → a slightly different waveform (stochastic duration).

This page runs the pipeline inside a Web Worker so synthesis never blocks the interface, encodes the returned Float32 samples to a WAV blob for the <audio> element, and draws the same samples as the waveform under "See inside". The English demo is the same call with a different checkpoint (mms-tts-eng). The Hindi tokenizer is native Devanagari (is_uroman:false), so Devanagari text is tokenised directly — no romanisation step.
