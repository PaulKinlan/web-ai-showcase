Phi-3.5-mini — a reasoning chat model that runs on your GPU

Microsoft's Phi-3.5-mini is a 3.8-billion-parameter instruct model punching well above its size on reasoning, maths, and structured output — and it runs entirely in your browser, compiled to WebGPU by MLC's WebLLM engine. You get streaming tokens, sampling controls, a system prompt, and full conversation history, with nothing uploaded and no API bill. It's a bigger, more capable step up from the 1B models: no WASM fallback, so it needs a real GPU and enough VRAM.

Why it matters

Phi-3.5-mini is small enough to run in a tab but strong enough to reason: it's competitive with much larger models on maths, code, and structured tasks. Running it on WebLLM makes that capability a client-side primitive — private by construction, free to run, and available offline once the weights are cached.

  • Private reasoning assistants — step-by-step help with maths, code, and analysis where the conversation never leaves the device.
  • Reliable structured output — JSON extraction and schema-shaped answers, using WebLLM's JSON mode, with a model good enough to get the fields right.
  • Longer-context work — summarise or answer over multi-paragraph documents within a 4k window, on-device.
  • On-device RAG — pair it with a local embedding model to answer grounded in a user's own documents (see the multi-model demo).

How the API works

WebLLM gives you an OpenAI-shaped chat API backed by a WebGPU engine. You create the engine (which downloads and compiles the weights), then stream a completion. This page uses the shared helpers in lib/webllm.js, running inside a Web Worker so the UI stays responsive: 

import { webGPUAdapterAvailable, createEngine, streamChat }
  from "/web-ai-showcase/lib/webllm.js";

// WebLLM is WebGPU-only — gate before doing anything (headless requestAdapter() → null).
if (!(await webGPUAdapterAvailable())) {
  showNeedsWebGPU();          // never a faked reply
} else {
  const engine = await createEngine({
    model: "Phi-3.5-mini-instruct-q4f16_1-MLC",
    onProgress: (p) => setProgress(p.text, p.progress),   // ~2.2 GB, cached after first load
  });

  const messages = [
    { role: "system", content: "You are a careful, concise assistant." },
    { role: "user",   content: "If a train travels 60 km in 45 minutes, what is its speed in km/h?" },
  ];

  const full = await streamChat(
    engine,
    { messages, temperature: 0.7, top_p: 0.95, max_tokens: 384 },
    (delta) => appendToken(delta),          // streams token-by-token on your GPU
  );
  // engine.interruptGenerate() stops decoding early; engine.runtimeStatsText() reports tok/s.
}

Everything here runs exactly that path in worker.js, and gates on a real WebGPU adapter first. No adapter → a labelled needs-WebGPU state, never a canned answer.
