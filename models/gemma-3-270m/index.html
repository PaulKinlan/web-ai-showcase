Gemma 3 270M — Google's smallest Gemma 3, in your browser

Gemma 3 270M is the smallest model in Google's 2025 Gemma 3 family — a ~270M-parameter instruction-tuned chat model built to run anywhere. This page runs it entirely in your browser via Transformers.js: it streams tokens on WebGPU (dtype q4f16) when your device has an adapter, and falls back to the universal WebAssembly/CPU path (dtype fp32) otherwise. Chat with it, steer it with a system prompt and sampling controls, and watch the real probability distribution behind every token. Nothing is uploaded; there is no API key and no server.

Distinct from the other small chat models here: this is the Transformers.js build of the tiniest Gemma 3, so it runs on the WASM/CPU path with no WebGPU required — where the Gemma 3 1B (WebLLM) demo needs WebGPU. Because the gemma3_text class is newer than the repo's shared Transformers.js pin, this one demo loads Transformers.js 4.2.0 in its worker (an isolated version pin; every other page stays on 3.7.5).

Why fp32 on WASM?The quantized exports (q4, q8) use the GatherBlockQuantized embedding op, which ORT-Web only implements on the WebGPU EP; on the WASM EP they abort at session creation. fp16/q4f16 likewise need WebGPU's fp16 kernels. So on the CPU path fp32 is the only export that runs — larger, but real coherent output. (Verified in headless Chrome.)

Why it matters

Gemma 3 270M is the "runs anywhere" tier of a serious open family from Google. Because it is so small, it makes client-side language a universal primitive: it works on the locked-down work laptop, the older phone, the kiosk with the GPU blocklisted — on WebGPU it's a ~200 MB fast download, and even with no GPU at all it still runs (on the fp32 CPU path). Private, offline-capable once cached, free to run, no key. Its real strength is as a task-specific base: tiny models like this are cheap to fine-tune for one narrow job (classification, extraction, formatting) and then embed directly in a page.

  • On-device drafting — short summaries, rewrites, and replies with nothing sent to a server.
  • Structured extraction — turn messy text into clean JSON for a form or API, entirely in the page.
  • Universal reach — WebGPU when available, WASM/CPU when not; if the browser runs WebAssembly, it runs.
  • On-device RAG — pair it with a local embedding model for grounded answers over a user's own documents (see the multi-model demo).

How the API works

It's the standard Transformers.js text-generation pipeline — pass a messages array (the pipeline applies Gemma's chat template), a TextStreamer for token-by-token output, and your sampling settings. The only device-specific choice is device + dtype, which this page picks by probing for a real WebGPU adapter. Note the pinned version — the gemma3_text class needs Transformers.js ≥ 4.2.0: 

import { pipeline, TextStreamer } from "@huggingface/transformers"; // v4.2.0+

// Probe for a real adapter — WebGPU (q4f16) fast path, else universal WASM (fp32) on the CPU.
const adapter = navigator.gpu && await navigator.gpu.requestAdapter().catch(() => null);
const [device, dtype] = adapter ? ["webgpu", "q4f16"] : ["wasm", "fp32"];

const generator = await pipeline(
  "text-generation", "onnx-community/gemma-3-270m-it-ONNX", { device, dtype },
);

const messages = [
  { role: "system", content: "You are a helpful, concise assistant." },
  { role: "user", content: "Name three primary colors." },
];

const streamer = new TextStreamer(generator.tokenizer,
  { skip_prompt: true, skip_special_tokens: true, callback_function: (t) => append(t) });

await generator(messages, {
  max_new_tokens: 200, do_sample: true, temperature: 0.7, top_p: 0.9, streamer,
});

The "see inside" panel runs one raw forward pass — generator.model(generator.tokenizer(prompt)) — reads the last position's logits, softmaxes them, and shows the real top-k. Everything runs in a Web Worker so generation never blocks the page; the loader auto-inits an already-cached copy and never silently re-downloads.
