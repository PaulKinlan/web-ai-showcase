OLMo-2 1B — a fully-open model on your GPU

Almost every "open" LLM you can name is open weights only: you get the finished model and nothing about how it was made. OLMo-2, from Ai2 (the Allen Institute for AI), is different — it's fully open. The training data (Dolma / Dolmino), the training code (OLMo-core), every intermediate checkpoint, and the training logs are all published under Apache-2.0. It's a reproducible recipe, not a black box. This page runs the 1B Instruct build entirely in your browser on MLC's WebLLM engine: streaming tokens, sampling controls, a real system prompt, nothing uploaded and no API bill. WebGPU only — an ~845 MB download, cached after first load.

How it differs from the other chat pages: OLMo-2 is the catalogue's only fully-open model — a distinct Ai2 OLMo architecture whose entire training recipe is public, separate from the open-weights Gemma-family (Gemma 3 1B), Qwen (Qwen3 1.7B), Mistral (Mistral 7B) and Llama-family models. Same job — a private, on-device chat — but the one you could rebuild from scratch.

Why it matters

For most products the practical win of a local LLM is the same regardless of provenance: private, free-to-run, offline-capable chat with no server or API bill. But a fully open model unlocks things an open-weights model can't — because you can see and change how it was built.

  • Reproducible & auditable — the training data and code are public, so you can audit what went in, check for contamination, and reproduce or continue pre-training. Essential for regulated, research, and safety-critical work.
  • Continued pre-training on your domain — start from a published intermediate checkpoint and keep training on your own corpus with the same OLMo-core recipe, instead of guessing at a closed process.
  • Private on-device assistants — the same benefit as any local chat model: sensitive conversations never leave the device, and it runs offline once cached.
  • On-device RAG — pair it with a local embedding model for grounded answers over your own documents, fully client-side (see the multi-model demo).

How the API works

WebLLM gives you an OpenAI-shaped chat API backed by a WebGPU engine. You create the engine (which downloads and compiles the weights), then stream a completion. OLMo-2 Instruct has a real system role, so no folding is needed — you pass system / user / assistant turns directly: 

import { webGPUAdapterAvailable, createEngine }
  from "/web-ai-showcase/lib/webllm.js";

// WebLLM is WebGPU-only — gate before doing anything (headless requestAdapter() → null).
if (!(await webGPUAdapterAvailable())) {
  showNeedsWebGPU();          // never a faked reply
} else {
  const engine = await createEngine({
    model: "OLMo-2-0425-1B-Instruct-q4f16_1-MLC",
    onProgress: (p) => setProgress(p.text, p.progress),   // ~845 MB, cached after first load
  });

  const messages = [
    { role: "system", content: "You are a concise assistant." },
    { role: "user", content: "Explain what makes OLMo-2 'fully open'." },
  ];

  const stream = await engine.chat.completions.create({
    messages, temperature: 0.7, top_p: 0.9, max_tokens: 512, stream: true,
  });
  for await (const chunk of stream) {
    const d = chunk.choices[0].delta;
    if (d.content) appendAnswer(d.content);      // stream the answer token by token
  }
  // engine.interruptGenerate() stops decoding early; engine.runtimeStatsText() reports tok/s.
}

Everything here runs exactly that path in worker.js (off the main thread), and gates on a real WebGPU adapter first. No adapter → a labelled needs-WebGPU state, never a canned answer.
