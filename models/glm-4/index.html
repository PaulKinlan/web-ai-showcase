GLM-Edge 1.5B — Zhipu's bilingual GLM-4 model, in a tab

GLM-Edge-1.5B-Chat is the browser-runnable member of Zhipu AI's GLM family — the same glm architecture as GLM-4, shrunk to 1.5 billion parameters for on-device use. It's genuinely bilingual: ask in English or Chinese (中文) and it answers in kind, or translates between them. It streams tokens on your own GPU with sampling controls, and you can open the hood on the real next-token probability distribution. No server, no API key, nothing uploaded.

How it differs from the other chat pages: it's a GLM (Zhipu) architecture, distinct from the Qwen (Qwen2.5 0.5B), Llama (Llama 3.2 1B) and SmolLM (SmolLM2) models here — and it's built for bilingual EN/ZH work out of one small model.

Why it matters

A bilingual instruct model that fits in a browser tab turns cross-language work into a client-side primitive: private, offline-capable, and free to run. GLM-Edge is Zhipu's answer to on-device GLM — the same family that powers GLM-4, sized down to reach real hardware. No inference bill, no data leaving the device, no per-language service.

  • Bilingual private chat — an assistant that handles English and Chinese equally, without sending a keystroke to a server.
  • On-device translation & localisation — draft, translate, and adapt copy between EN and ZH inside the page.
  • Structured extraction — turn messy text (in either language) into clean JSON for a form or API.
  • Offline & edge — kiosks and field apps for bilingual audiences that must work with no network.

How the API works

GLM-Edge uses the standard text-generation pipeline. Pass a messages array (the pipeline applies GLM's chat template for you), a TextStreamer for token-by-token output, and your sampling settings: 

import { pipeline, TextStreamer } from "@huggingface/transformers";

const generator = await pipeline(
  "text-generation",
  "onnx-community/glm-edge-1.5b-chat-ONNX",
  { dtype: "q4f16", device: "webgpu" },   // WebGPU path (~1 GB); WASM q4 is a slow opt-in
);

const messages = [
  { role: "system", content: "You are a helpful bilingual assistant." },
  { role: "user", content: "用一句话解释什么是 WebGPU。" },   // ask in Chinese…
];

const streamer = new TextStreamer(generator.tokenizer,
  { skip_prompt: true, skip_special_tokens: true, callback_function: (t) => append(t) });

await generator(messages, {
  max_new_tokens: 256, do_sample: true, temperature: 0.7, top_p: 0.9, streamer,
});   // …and it answers in Chinese

The "see inside" panel runs one raw forward pass — generator.model(generator.tokenizer(prompt)) — reads the last position's logits, softmaxes them, and shows the real top-k. Everything runs in a Web Worker and gates on a real WebGPU adapter first; with none, it shows a labelled needs-WebGPU state (plus an opt-in WASM path) instead of faking tokens.
