Jina reranker (tiny) — precision search, small enough to ship on-device

A cross-encoder reads a query and a passage together and outputs one relevance score — far more accurate than comparing two separate embeddings. Most cross-encoders are big. This one is tiny: 4 Transformer layers, ~33 MB quantized, built by Jina AI specifically to be fast enough to rerank on a phone or in a tab. Give it a query and some candidates, watch it re-sort them, and look at the real per-pair latency — the whole point is that precision no longer means a slow, server-side model. Nothing leaves your device.

Why it matters

Rerankers are the precision stage of modern search. A fast retriever (a keyword index or embeddings) casts a wide net and returns, say, the top 50 candidates; a cross-encoder then reads each candidate against the query and reorders them with far higher accuracy. The catch has always been cost — a cross-encoder runs the full model on every (query, passage) pair, so teams push it to a server. A tiny reranker changes the economics: at ~33 MB and a few milliseconds per pair on a CPU, the precise step can run on the device, inside a private RAG loop, with no inference bill and no data leaving the browser.

  • On-device RAG — rerank the retriever's shortlist locally before feeding the best chunk to a local LLM.
  • Instant client-side search — reorder autocomplete or in-page results as the user types, no round-trip.
  • Edge / offline — ship precise ranking to a PWA that works with no network at all.
  • Cheap answer selection — pick the best reply from candidates without a hosted reranker.

Basics

Rank a handful of passages against a query by cross-encoder relevance.

Wild

Hard negatives: distractors stuffed with the query's words that answer the wrong thing.

Multi-model

MiniLM embeddings retrieve, the tiny Jina reranker reorders — the classic two-stage pipeline.

Contrast with the larger BGE-reranker (multilingual, ~270 MB) and the MS MARCO MiniLM reranker: same cross-encoder idea, but this one trades reach for size and speed.

How the API works

A cross-encoder scores query–passage pairs; we read the raw logit off the model directly:

import { AutoTokenizer, AutoModelForSequenceClassification } from "@huggingface/transformers";

const model_id = "jinaai/jina-reranker-v1-tiny-en";
const tokenizer = await AutoTokenizer.from_pretrained(model_id);
const model = await AutoModelForSequenceClassification.from_pretrained(model_id, { dtype: "q8" });

// Encode the query with each passage AS A PAIR — [CLS] query [SEP] passage [SEP]:
const inputs = tokenizer(
  passages.map(() => query),
  { text_pair: passages, padding: true, truncation: true },
);
const { logits } = await model(inputs); // one relevance logit per pair — higher = more relevant
// sort passages by logits.data — that's your reranking, and at ~33 MB it's fast enough on a CPU
// to run right inside the browser

This page loads the model through the shared loader as a text-classification pipeline, then reuses pipe.model + pipe.tokenizer to read the raw logits (the pipeline would squash them through a sigmoid). All inference runs in a Web Worker so ranking never janks the page.
