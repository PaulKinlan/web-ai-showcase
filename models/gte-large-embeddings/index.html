GTE-large — the big embedder, 1024 numbers per sentence

GTE-large is the heavyweight sibling of GTE-small: the same recipe — multi-stage contrastive training, mean pooling, and no instruction prefix — but a BERT-large backbone (~335M parameters) that emits a 1024-dimensional sentence vector instead of 384. Those extra parameters and dimensions buy a higher MTEB retrieval score and sharper separation on hard, near-synonymous pairs — the price is a much bigger download (~337 MB) and slower per-embedding latency. Type a few lines to build a live similarity matrix, search by meaning, and watch GTE-large and GTE-small score the SAME pairs side by side. Nothing leaves your device.

See inside — the 1024-d vector, and what the extra size buys

The encoder emits one 1024-number vector per token. To get a single vector per sentence GTE mean-pools those token vectors (mask-aware) and L2-normalizes, so every sentence becomes a unit-length arrow and cosine similarity is a plain dot product. Pick a sentence to see its real vector and its pre-normalization magnitude — 1024 numbers, not 384.

Why it matters

Embeddings are the quiet workhorse behind search, recommendations, deduplication, clustering, and retrieval-augmented generation. GTE-large's pitch is quality: when a mis-ranked result is expensive — legal search, support deflection, dedup at scale — the bigger model's sharper separation earns its download. And it still runs entirely client-side, so the text never leaves the device. The honest tradeoff: reach for GTE-small when latency and download budget matter, GTE-large when ranking accuracy does.

  • High-stakes semantic search — rank a corpus by meaning where a wrong top hit costs you, entirely client-side.
  • Near-duplicate detection — catch "same thing, different words" pairs the small model scores as merely related.
  • Quality-first RAG retrieval — fetch the right snippet to feed a local LLM when precision beats speed.
  • Pick-your-tier embedding — ship small for the edge, large for the server-grade result, same API and pooling.

High-quality semantic search plus near-duplicate detection over your own corpus.

Wild

Cluster a mixed bag by meaning, and hunt hard pairs where GTE-large sees a distinction GTE-small misses.

Multi-model

GTE-large retrieves a shortlist by meaning, then a cross-encoder reranker re-scores the top hits.

How the API works

Feature-extraction plus the pooling discipline is the whole recipe — same as GTE-small, just a bigger model and 1024-d output, and still no prefix:

import { pipeline } from "@huggingface/transformers";

const extract = await pipeline(
  "feature-extraction",
  "Xenova/gte-large",
); // downloads + caches ~337 MB once

// GTE uses MEAN pooling and needs NO instruction prefix — embed query and doc identically:
const out = await extract(["how do I sign in?", "the login page rejects my password"], {
  pooling: "mean",
  normalize: true,
});
const [q, d] = out.tolist();     // each is a 1024-number unit vector

// unit-length vectors → cosine similarity is just the dot product:
const sim = q.reduce((s, x, i) => s + x * d[i], 0); // high — same meaning

This page asks the worker for the un-normalized pooled vectors so "See inside" can show the real pre-normalization magnitude, then normalizes them in JS. Inference runs in a Web Worker, so the matrix, projection and comparison never jank the page.
