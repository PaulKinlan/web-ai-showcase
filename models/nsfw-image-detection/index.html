NSFW detector — is this image safe to display?

A Vision Transformer fine-tuned to sort images into sfw (safe for work) and nsfw. Drop an image and it returns a probability for each and a plain verdict — safe to show, or hold for review. This is defensive content-safety tooling: an upload gate you can run before an image is ever displayed or stored. Everything happens on your device; the image is never uploaded.

See inside — score vs. policy

The model outputs two raw logits; a softmax turns them into two probabilities that sum to 100%. The model gives you the number; the threshold is your policy. A stricter threshold (e.g. 0.20) catches more borderline images but withholds more safe ones (false positives); a looser one (0.80) shows more but risks letting something through (false negatives). This is the moderation trade-off in one slider — there's no single right value, it depends on how costly each mistake is for your product.

Why it matters

Any product that lets users upload images inherits a moderation problem. A fast, private, offline safe/NSFW score is a first filter that runs before an image is stored, displayed, or sent anywhere — no API bill, no third party seeing user content, and a latency low enough to gate an upload in real time.

  • Upload gates — screen a profile photo or attachment client-side and block or blur it before it's posted.
  • Avatar & thumbnail screening — auto-approve the obviously-safe majority and route only the uncertain ones to human review.
  • Private-by-default safety — sensitive images get screened without ever leaving the user's device.
  • Cost control — put a free on-device filter in front of an expensive moderation API or human queue.

How the API works

The convenience path is two lines with Transformers.js:

import { pipeline } from "@huggingface/transformers";

const detect = await pipeline("image-classification", "AdamCodd/vit-base-nsfw-detector");
const out = await detect(imageURL, { top_k: 2 });
// → [{ label: "sfw", score: 0.98 }, { label: "nsfw", score: 0.02 }]

This page goes one level deeper for "See inside": instead of the pipeline it runs the model's processor then model() in a single forward pass, so it can read the raw two-value logits tensor and compute the softmax itself — then apply your threshold. Same weights, same result. Inference runs in a Web Worker so the page never janks.
